## Executive summary

Several files within `opencode/skills/` exhibit patterns that violate the bridge-burning policies, particularly "No defaults in runtime logic", "No fallbacks, period", "Error laundering", and "No optional critical dependencies." These patterns suppress failure visibility, making debugging difficult and encouraging a false sense of correctness.

## Candidate findings

### Finding 1: Error Laundering / Partial Success

- Pattern source: Error laundering
- Files / lines:
  - `opencode/skills/literature-review/scripts/lit_search.py:110` — `except Exception as e: return {"error": str(e), "data": []}`
  - `opencode/skills/literature-review/scripts/lit_search.py:172` — `except Exception as e: return {"error": str(e), "data": []}`
  - `opencode/skills/literature-review/scripts/lit_search.py:215` — `except Exception as e: return {"error": str(e), "data": []}`
  - `opencode/skills/literature-review/scripts/lit_search.py:345` — `except Exception as e: return {"error": str(e), "data": []}`
- Claim:
  The `search_s2`, `search_oa`, `search_cr`, and `search_pm` functions use a catch-all `except Exception` block to catch any error during an API request or parsing. They convert the failure into a dictionary returning `{"data": []}`.
- Why this matches the pattern:
  This perfectly matches the "Error laundering" pattern ("converting failures into logs, empty arrays, partial objects..."). It fails open, allowing the caller to keep executing on empty arrays rather than forcing it to handle an explicit failure state or abort.
- Why this is not merely a user design choice:
  Returning a generic error dictionary with an empty array `[]` on an unexpected exception is a defensive programming hack to appease the caller's structure.
- Suggested remediation direction:
  Remove the `except Exception:` block and allow unexpected failures to fail loudly. For expected network exceptions like timeouts or connection errors, catch specific exceptions and raise structured errors or let them propagate.
- Confidence:
  High
- What would falsify this:
  If the application consuming this script depends on swallowing all exceptions (including syntax errors or memory errors) to gracefully degrade (though this violates bespoke software principles).

### Finding 2: Fallback Chains and Optional Dependencies

- Pattern source: Optional critical dependencies & Fallback chains
- Files / lines:
  - `opencode/skills/caveman-compress/scripts/compress.py:88` — `except ImportError: pass # anthropic not installed, fall back to CLI`
- Claim:
  The `call_claude` function tries to import `anthropic` and use the Python client. If `anthropic` is not installed, it catches the `ImportError` and falls back to invoking the `claude` CLI via `subprocess`.
- Why this matches the pattern:
  This matches both "Fallback chains" ("try real API, else cached fake", "try configured command, else default command") and "Optional critical dependencies" ("Lets the app pretend required tools are optional").
- Why this is not merely a user design choice:
  In bespoke software, dependencies should be explicitly declared and required. If the script requires `anthropic`, the doctor/setup should install it, and runtime should assume it's available. If it uses the CLI, it should just use the CLI.
- Suggested remediation direction:
  Remove the fallback. Pick one method (either `anthropic` client or `claude` CLI), make it the explicit requirement, and crash loudly if it's not available.
- Confidence:
  High
- What would falsify this:
  If the user explicitly requested the fallback behavior as a design choice to allow running in environments where they specifically do not want to install Python packages but have the CLI.

### Finding 3: Runtime Defaults and Silenced Exceptions

- Pattern source: Runtime defaults & Error laundering
- Files / lines:
  - `opencode/skills/google-workspace/scripts/google_api.py:74` — `except Exception: return list(SCOPES)`
  - `opencode/skills/google-workspace/scripts/setup.py:78` — `except Exception: return {}`
- Claim:
  The functions `_stored_token_scopes` and `_load_token_payload` use a catch-all `except Exception` block when reading and parsing a JSON file. If any error occurs (file not found, permission denied, invalid JSON), it returns a default value (`list(SCOPES)` or `{}`).
- Why this matches the pattern:
  This matches "Runtime defaults" and "Pointless catching" / "Error laundering." The code swallows all errors and replaces them with a default, meaning malformed configurations or corrupted files are silently ignored.
- Why this is not merely a user design choice:
  Silently ignoring corrupted configuration state is not a desired behavior; it masks the root cause of failure.
- Suggested remediation direction:
  Instead of a broad `except Exception:`, catch only the explicitly expected `FileNotFoundError` if the file might legitimately not exist yet. If the file exists but contains invalid JSON, the program should crash loudly rather than returning an empty dictionary.
- Confidence:
  High
- What would falsify this:
  If the file corruption is explicitly expected and the user specifically requested a graceful recreation without user intervention.

### Finding 4: Banned Exception Swallowing

- Pattern source: Banned try/catch/except blocks
- Files / lines:
  - `opencode/skills/reading-transcripts/scripts/list_all_sessions.py:67` — `except Exception: return 0`
  - `opencode/skills/reading-transcripts/scripts/list_all_sessions.py:180` — `except Exception: pass`
- Claim:
  The transcript listing script wraps database queries (`sqlite3`) in broad `except Exception` blocks and returns `0` or `pass`es on failure.
- Why this matches the pattern:
  This matches the "Banned: try/except fallback" and "catch-and-log continuation" (or rather catch-and-ignore). Database failures (e.g., locked database, malformed schema) are silently ignored.
- Why this is not merely a user design choice:
  It converts failed work into partial success objects, hiding failures from the user.
- Suggested remediation direction:
  Remove the `try/except` block and allow database connection and querying errors to raise and crash loudly.
- Confidence:
  High
- What would falsify this:
  If the database is known to be routinely locked by other processes and the script is intended to be a best-effort background monitor where failures are explicitly allowed to be dropped.
