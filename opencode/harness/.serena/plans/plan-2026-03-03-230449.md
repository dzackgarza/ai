# Plan: Clean Developer-Facing `opx` CLI (Session Automation Harness)

## Goals
- Provide a coherent, stable CLI for plugin developers to automate OpenCode sessions.
- Support primitive session management: create/list/send/abort/delete/wait.
- Support attach/resume semantics: `send` targets an existing session; if the session is busy, queue or wait deterministically.
- Provide reliable internal status (idle/working, current tool, last step/error) without parsing human transcripts.
- Keep probes/internal utilities out of the main surface (move under `opx debug ...`).

## Current State (Observed)
- Core session commands already exist in `src/cli.ts`: `new`, `list`, `send`, `wait`, `abort`, `delete`, `tail`, `messages`, plus error/trace helpers.
- Internal status is not truly implemented; `status` currently just lists sessions.
- SDK types expose:
  - SSE events: at least `message.updated`, `message.part.updated`, `session.updated`, `session.idle`, `session.error`, `session.deleted`.
  - Message parts include `tool` parts with `state.status` in `{pending,running,completed,error}` and `step-start`/`step-finish` parts.
- Async prompt API is currently called via raw fetch to `/session/:id/prompt_async`.

## Design Decisions (Proposed)
1. **Command surface re-org**
   - Public surface: `opx session ...` (primary), plus a small set of convenience aliases for backward compatibility.
   - Internal/probe surface: `opx debug ...`.

2. **Status is derived from server events + message parts**
   - Do not parse transcripts.
   - Compute `working/idle` as a derived state from:
     - Most recent `tool` part state (`running` => working).
     - Presence of `step-start` without a later `step-finish` for the same message (heuristic: working).
     - `session.idle` event (authoritative signal of idle when received).
     - `assistant` message with `completed` timestamp missing (working).
   - Provide `current_tool` and `last_tool` from the latest `tool` part(s).
   - Provide `last_error` from `assistant.error` or `session.error`.

3. **Attach/resume + queueing behavior**
   - `opx session send --session <id>` always targets the given session; no implicit `new`.
   - Preferred mechanism: use async prompting and then optionally `wait`.
     - This avoids client-side timeouts and supports background processing.
   - Queueing strategy:
     - First attempt prompt (async).
     - If server rejects due to “busy/locked”, fallback to: subscribe to events, wait for `session.idle`, retry prompt.
     - If server accepts while busy (server-side queue), no special logic needed.

4. **Transcripts are human-read artifacts**
   - Keep `tail` functionality, but label it explicitly as “messages transcript view”.
   - Add a separate `session inspect` command that prints: derived status + recent tool/step + optional tail.

## CLI Spec (Target)
### Public
- `opx session list` (table or JSON via `--json`)
- `opx session new [--title T] [--prompt P] [--agent A] [--model provider/model] [--async]`
- `opx session send --session ID --prompt P [--agent A] [--model provider/model] [--async] [--no-reply] [--wait] [--timeout SEC]`
  - `--wait` means wait until idle or until a predicate is matched.
- `opx session wait --session ID [--contains TEXT] [--timeout SEC]`
  - Implement using SSE if possible; fallback to polling messages.
- `opx session status --session ID [--watch] [--json]`
  - Derived status: `{state: idle|working|error?, currentTool?, lastStepTime?, lastError?}`
- `opx session tail --session ID [--lines N]` (message transcript view)
- `opx session abort --session ID`
- `opx session delete --session ID`
- `opx session inspect --session ID [--lines N] [--json]` (status + tail)

### Debug (move existing probes)
- `opx debug probe-async-command ...`
- `opx debug probe-async-subagent ...`
- `opx debug probe-limit ...`
- `opx debug probe-limit-known ...`
- `opx debug probe-limit-trace ...`
- `opx debug trace --session ...`

### Compatibility aliases (optional)
- Keep old top-level commands (`new`, `send`, `tail`, etc.) as thin wrappers that print a deprecation warning and dispatch to `session ...`.

## Implementation Steps
1. **Checkpoint + hygiene (pre-edit)**
   - Inspect `git status` and ensure there is a clean baseline commit for harness-only work.
   - If repo is dirty with unrelated changes, avoid staging them; isolate harness edits.

2. **Refactor CLI structure (no behavior changes initially)**
   - Extract command handlers into modules (e.g. `src/commands/session/*.ts`, `src/commands/debug/*.ts`) or keep in one file but with namespacing.
   - Centralize arg parsing and output formatting (`--json` support).

3. **Implement `session status` (real status)**
   - Add a status builder that:
     - Fetches session messages, extracts recent `tool` parts + assistant message completion/errors.
     - Optionally opens SSE stream for a short window to capture latest `session.updated`, `message.part.updated`, `session.error`, `session.idle`.
   - Return derived `idle/working` + `currentTool` if any tool is running.

4. **Upgrade `wait` to event-driven**
   - Subscribe to SSE, filter by `sessionID`, wait for:
     - `session.idle` (no `--contains` case), or
     - assistant text containing substring, using `message.part.updated` text parts.
   - Keep polling fallback when SSE fails.

5. **Implement queueing semantics in `send`**
   - Default to async prompt.
   - If async prompt fails with a busy/locked response (detect via HTTP status/body), wait for `session.idle` and retry once (or with bounded retries).

6. **Move probes under `debug`**
   - Re-home probe commands; keep behavior identical.

7. **Docs**
   - Update `README.md` to present the new command surface and the “status is derived, transcript is read-only view” model.

8. **Validation**
   - `bun run check`
   - Smoke: start `opencode serve` and exercise:
     - `session new`, `session send --async --wait`, `session status --watch`, `session tail`, `session abort`, `session delete`.
   - Confirm `session status` shows running tool + transitions to idle.

## Open Question (Need User Input)
- Should we keep backward-compatible top-level commands (`opx new`, `opx send`, etc.) as aliases, or make `opx session ...` the only public surface?
  - Recommended default: keep aliases for 1 release with a deprecation warning to reduce churn.
