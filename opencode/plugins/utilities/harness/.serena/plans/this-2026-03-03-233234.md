# Plan: Tester-First `opx` Harness (Run/Resume + Transcript Artifacts)

This plan replaces the earlier "session send/async/no-reply"-centric design. It is explicitly oriented around what testers validate: the human-visible transcript text.

## 1) Product Requirements (Restated)

### Primary workflows
1) **One-shot (default)**
- Behavior: `user prompt -> agent actions -> idle -> transcript -> delete session`.
- Must delete session by default, but only **after** transcript is generated.
- Test output should be the **transcript text** (what users see), not machine JSON.

2) **One-shot with background continuation window**
- Behavior: `prompt -> agent actions -> idle -> further events -> further agent actions -> idle -> transcript -> delete session`.
- Must support:
  - overall timeout (entire chain)
  - "linger" window after first idle (keep watching for new activity)

3) **Resume (multi-turn) against existing session**
- Behavior: same as (1)/(2), but starting from a saved session.
- If the session is still busy, the tester needs tools to:
  - introspect what it’s doing
  - wait until it is safe to send the next prompt

### Introspection/management
- Determine if session is working vs idle.
- Determine what it is doing now (current tool), last thing it did (last tool), and last error.
- View transcript (tail or full) as a human-facing artifact.
- Abort and delete for cleanup.

### Provider unavailability signal
- Tests must clearly distinguish:
  - genuine test failures from
  - provider unavailable due to rate-limit/quota/auth.
- Must produce a distinct exit code for provider-unavailable.

### Constraints
- **Do not parse transcripts** to compute status/errors.
- Status/errors come from internal session/message/event state.
- Transcript is produced via the existing transcript tool/script that takes a **session ID**.

## 2) Public CLI Surface (What testers use)

### 2.1 `opx run` (create session automatically)
`opx run --prompt <text> [--agent <agent>] [--model <provider/model>] [--timeout <sec>] [--linger <sec>] [--keep] [--transcript-out <path>] [--no-print]`

Defaults
- `--timeout`: 180 (or keep current REQUEST_TIMEOUT_MS as baseline, but expose as seconds)
- `--linger`: 0 (disabled)
- `--keep`: false
- Transcript output path default: `/tmp/transcript-<session_id>.txt`
- Printing default: **print transcript to stdout** (unless `--no-print`)

Algorithm
1. Create new session with title `opx-run:<epoch>` (optionally include model/agent for debugging).
2. Send prompt to session (sync or async under the hood is an implementation detail; public UX is synchronous run-to-idle).
3. Wait until the session reaches idle (see waiting logic).
4. If `--linger > 0`: keep watching for renewed activity; stop when:
   - session is idle, and
   - no activity events for `linger` seconds,
   bounded by `--timeout` overall.
5. Generate transcript file by calling transcript script with session ID.
6. Print transcript text to stdout (unless `--no-print`).
7. Print `Transcript saved to: <path>` and (if kept) `Session kept: <id>`.
8. Delete session unless `--keep`.

Failure rules
- If prompt send fails in a provider-unavailable way: still attempt transcript generation (if possible), then exit code 2.
- If waiting times out: still generate transcript (so user sees what happened), keep session by default unless user explicitly requested delete.
- If transcript generation fails: keep session, print session ID, exit non-zero.

Exit codes
- `0`: completed run, no provider-unavailable detected, no timeout.
- `1`: run failed (timeout, unknown error, aborted, etc.).
- `2`: provider unavailable (rate_limit/quota/auth) detected.

### 2.2 `opx resume` (reuse an existing session)
`opx resume --session <ses_id> --prompt <text> [--agent <agent>] [--model <provider/model>] [--timeout <sec>] [--linger <sec>] [--keep] [--transcript-out <path>] [--no-print]`

Algorithm
1. Compute current status (working/idle + current tool + last error).
2. If working: wait for idle (bounded by `--timeout`).
3. Send new prompt.
4. Apply same wait/linger rules as `run`.
5. Generate transcript; print transcript + saved path.
6. Delete unless `--keep`.

Busy-session behavior (key)
- If session is busy, tester gets:
  - immediate status line explaining current tool and how long it’s been running (if available)
  - then we wait.

### 2.3 `opx session ...` (inspection + cleanup)
- `opx session list`
  - prints `id\ttitle\tupdated`.
- `opx session status --session <ses_id> [--watch] [--interval <sec>]`
  - One-shot status or watch mode.
  - Output is human-readable.
- `opx session transcript --session <ses_id> [--lines <n>] [--out <path>]`
  - Always uses transcript script by session ID.
  - If `--lines`, tail the resulting file for convenience.
- `opx session abort --session <ses_id>`
- `opx session delete --session <ses_id>`

### 2.4 `opx provider check ...` (explicit provider signal)
- `opx provider check --provider <key> [--timeout <sec>]`
  - Uses strict known-pattern checks (existing `known_limit_patterns.json` machinery).
  - Output is human-readable, exit code `2` for unavailable.

## 4) Transcript Integration (Canonical output)

### Mechanism
- Invoke the transcript parser script with session ID and write its output to a text file.
- This matches existing tooling:
  - script: `parse_opencode_log.py <session_id> > /tmp/transcript-<id>.txt`

### Configuration
- Env var: `OPX_TRANSCRIPT_SCRIPT` overrides default script location.
- Default: reuse the same path used by `opencode/plugins/read-transcript.ts`.

### File path rules
- Default: `/tmp/transcript-<session_id>.txt`
- If `--transcript-out` given:
  - allow explicit path, including templating token `{session_id}` (e.g. `--transcript-out /tmp/opx-{session_id}.txt`).

### Output rules
- Always print the transcript path.
- Default prints transcript body to stdout; `--no-print` prints only path + status footer.

## 5) Waiting + Linger Logic (deterministic run semantics)

### Event-driven waiting (preferred)
- Subscribe via `client.event.subscribe`.
- Filter for relevant events for the target session:
  - `session.idle` (authoritative idle signal)
  - `session.error` (capture error classification)
  - `message.updated` and `message.part.updated` (activity)
  - `session.updated` (activity)

### Activity definition
Any of the above events for the session counts as activity and resets a `lastActivityAt` timestamp.

### Idle determination
Maintain a derived state:
- `idleSeenAfterPrompt`: set when `session.idle` arrives after the prompt send timestamp.
- `toolRunning`: true if latest observed message parts contain a tool part with `state.status="running"`.
- Consider session *idle now* if:
  - `idleSeenAfterPrompt` is true, and
  - `toolRunning` is false.

### Linger
If `linger > 0`:
- After reaching idle, keep listening.
- Exit when:
  - session is idle now, and
  - `now - lastActivityAt >= lingerSeconds`.

### Fallback if SSE fails
- Poll `client.session.messages({id})` periodically.
- Infer `toolRunning` by scanning message parts for tool parts in running state.
- Infer idle by stability of state over multiple polls + lack of toolRunning.
- Still bounded by timeout.

## 6) Status Computation (for `session status` and for resume)

Input sources (internal, not transcript)
- `client.session.messages({id})` for current message parts.
- Optional: short-lived SSE subscription to get the latest `session.idle`/`session.error` quickly.

Outputs
- `state`: `working|idle`.
- `currentTool`: newest tool part with `state.status="running"` (tool name + start time).
- `lastTool`: newest tool part with `state.status in {completed,error}` (tool name + end time + status).
- `lastError`: latest `assistant.error` or `session.error`, classified.

Error classification
- Use assistant error objects when present.
- Map to categories:
  - rate_limit, quota, auth, aborted, other
- Also support known-pattern evidence via provider check as a separate command (not required for status).

## 7) Provider Unavailability Handling

### Within run/resume
- If we observe provider-unavailable classification during the run:
  - Append a short footer line after transcript path, e.g.:
    - `Provider unavailable: rate_limit (providerID/modelID)`
  - Exit with code 2.

### Explicit preflight
- `opx provider check --provider <key>` remains strict and deterministic for known cases.

## 8) Session Lifecycle Rules
- Default: delete session after transcript generation succeeds.
- `--keep`: never delete.
- If transcript generation fails: keep session.
- If timeout occurs:
  - keep session by default unless `--keep=false` and user explicitly requested `--delete-on-timeout` (optional future flag).
  - Generate transcript anyway.
